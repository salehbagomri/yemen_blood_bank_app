## 🎯 السياق (Context)

تم تحويل التطبيق من نطاق **محافظة واحدة (المهرة)** إلى **اليمن كاملاً (22 محافظة، 224+ مديرية)**. التحويل المنجَز حتى الآن كان **سطحياً** (إعادة تسمية + إضافة خريطة المديريات + قوائم منسدلة متتالية)، لكن **البنية الأعمق ما زالت تفترض نطاقاً واحداً**. هذه الخطة تكمل التحويل الجوهري على ثلاثة محاور متوازنة: **تجربة المستخدم العادي**، **التوسع الخلفي والأداء**، و**الحوكمة الجغرافية**.

### الفجوات الجوهرية المكتشفة (من الاستكشاف)
1. **لا حوكمة جغرافية:** المستشفى **ترى كل متبرعي اليمن** (لا معنى لها محلياً) — `dashboard_provider.dart:26-42` و `manage_donors_hospital_screen.dart:540-544` لا يوجد بهما أي تقييد بالمحافظة. لا حقل يربط حساب المستشفى بمحافظتها.
2. **الموقع مدمج في حقل واحد** `district = "محافظة - مديرية"` — لا عمود محافظة مستقل، لا فهرسة، إحصائيات بـ 224+ مفتاح.
3. **إحصائيات لا تتوسع:** `statistics_service.dart:24-108` و `donor_service.dart:322-341,381-397` تجلب **كل المتبرعين** وتعدّهم في الجهاز (ينهار الأداء مع آلاف السجلات).
4. **تضارب الفلترة المحلية:** `manage_donors_screen.dart:552` يستخدم `startsWith` (صحيح)، بينما `advanced_search_screen.dart:230` و `donor_provider.dart:102` يستخدمان `==` (خطأ — لا يجد مديريات المحافظة في وضع عدم الاتصال).
5. **احتكاك UX للمستخدم العادي:** غموض صيغة الهاتف، قوائم منسدلة طويلة جداً، لا دليل أول مرة، المديرية إجبارية في الإضافة.

### القرارات المعتمدة من المستخدم
- **الحوكمة:** مستشفى مقيّدة بمحافظتها + أدمن عام يرى كل اليمن.
- **البيانات:** إضافة عمود `governorate` منفصل مع فهرسة و migration للملء.
- **الأولوية:** توازن شامل (UX + خلفي + حوكمة).
- **المتبرع:** يبقى بلا حساب (أقصى بساطة) — الإدارة/المستشفى تدير التحديثات.

## 🗺️ السيناريو الجديد (New Scenarios & Data Flow)

### أ. المستخدم العادي — طالب الدم (Blood Seeker)
> الأبسط على الإطلاق — لا يتغير جوهرياً، بل يُبسَّط أكثر.
```
يفتح التطبيق ← الرئيسية ← "بحث عن متبرع"
  1. يختار المحافظة (قائمة قابلة للبحث بالكتابة)
  2. يختار المديرية — أو يترك "كل المديريات" (افتراضي) للبحث بالمحافظة كاملة
  3. يضغط فصيلة الدم (chip واحد)
  ← نتائج فورية تُعرض من الأقرب جغرافياً، مع عدّاد "وُجد 15 متبرعاً"
  ← يضغط بطاقة ← اتصال مباشر / واتساب برسالة جاهزة
```
**تدفق البيانات:** الشاشة → `DonorProvider.searchDonors(governorate, district?, bloodType)` → `DonorService` → RPC `search_donors(p_governorate, p_district, p_blood_type, p_available_only)` على Supabase (مفهرس) → النتائج تُخزَّن في Hive للعمل دون اتصال.

### ب. المستخدم العادي — المتبرع (Donor)
> يبقى **بلا حساب**. تسجيل فوري.
```
الرئيسية ← "سجّل كمتبرع"
  1. الاسم
  2. رقم الجوال (مع توضيح بصري: 9 أرقام تبدأ بـ 7، بادئة 967+ تلقائية)
  3. فصيلة الدم (chips)
  4. المحافظة ← المديرية (متتالية، قابلة للبحث)
  5. العمر + الجنس
  6. (اختياري) آخر تبرع / ملاحظات
  ← حفظ ← يُخزَّن مع governorate + district منفصلين ← يظهر فوراً في البحث
```

### ج. قسم الإدارة — مستشفى مقيّدة بمحافظتها (الجوهر الجديد)
```
المستشفى تسجل دخول ← لوحة تحكم تعرض إحصائيات محافظتها فقط
  - "متبرعو محافظة حضرموت: 320 | متاح: 210 | موقوف: 40"
  - إدارة المتبرعين ← قائمة محافظتها فقط (RLS يفرضها خادمياً)
  - التقارير ← مقيّدة بالمحافظة
  - إضافة متبرع ← المحافظة مثبّتة على محافظة المستشفى تلقائياً
```
**تدفق البيانات:** حساب المستشفى يحمل `governorate` ← كل استعلاماته تُفلتر بـ `governorate = hospital.governorate` على مستوى الخدمة **و** على مستوى RLS في Supabase.

### د. قسم الإدارة — الأدمن العام
```
الأدمن يسجل دخول ← لوحة وطنية شاملة
  - إحصائيات كل اليمن + فلتر اختياري بالمحافظة
  - إدارة كل المستشفيات (وربط كل مستشفى بمحافظتها)
  - إدارة كل المتبرعين + فلترة بالمحافظة/المديرية
  - مراجعة البلاغات على مستوى الدولة
```

### الهيكلة المعمارية (طبقية، بدون تغيير النمط)
```
Screen → Provider → Service → Supabase (RPC + RLS)
                            ↘ Hive Cache (offline)
الجديد: عمود governorate مفهرس + RLS للتقييد الجغرافي + إحصائيات GROUP BY خادمية
```
